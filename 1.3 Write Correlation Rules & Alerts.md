## 🎯 Objective

Design and implement **SPL-based correlation searches**, detection rules, and alerting mechanisms in Splunk to identify security threats such as brute-force attacks and lateral movement.

---

# 📌 STEP 1: Brute-Force Detection Rule

---

## 🔐 Detection Logic

This rule detects repeated authentication failures from a single source IP within a short time window.

---

## 🧠 SPL Query

```spl id="bf1"
index=security_logs EventCode=4625
| bucket _time span=5m
| stats count by src_ip, _time
| where count > 10
```

---

## ⚙️ Save as Alert

Configure:

* Alert Title: **Brute Force Detected**
* Trigger Condition: `Number of Results > 0`

---

## ⏱️ Throttling Configuration

* Suppress repeated alerts:

  * Frequency: Once every **10 minutes**
  * Scope: Per `src_ip`

> ⚠️ Throttling is essential to prevent alert flooding and SOC fatigue.

---

# 📌 STEP 2: Lateral Movement Detection

---

## 🧠 Detection Logic

Detects a single user authenticating across multiple hosts in a short time frame, indicating potential lateral movement.

---

## 🧾 SPL Query

```spl id="lm1"
index=security_logs EventCode=4624 Logon_Type=3
| stats dc(ComputerName) as hosts by user
| where hosts > 5
```

---

## 🚨 Alert Condition

* Trigger when a user logs into **more than 5 distinct hosts**

---

## 🧩 Correlation Search Requirements

* Save as **CIM-compliant correlation search**
* Ensure field normalization for:

  * `user`
  * `host / ComputerName`
  * `authentication events`

# 📌 STEP 4: Tune & Suppress False Positives

---

## 🧹 False Positive Management

Identify trusted sources such as:

* IT admin jump servers
* Security scanners
* Monitoring tools

---

## 📋 Lookup-Based Whitelisting

Create lookup file:

```text id="wl1"
whitelist_ips.csv
```

---

## 🔍 Updated SPL with Lookup

```spl id="wl2"
index=security_logs EventCode=4625
[ inputlookup whitelist_ips.csv
| rename ip as src_ip
| table src_ip ]
| stats count
```

---

## 🚫 Exclusion-Based Filtering

Use exclusion logic:

```spl id="wl3"
... NOT [ inputlookup whitelist_ips.csv | fields ip ]
```

# ⚠️ BEST PRACTICE GUIDELINES

---

## 🧠 Detection Engineering

* Use **Splunk ESCU (Enterprise Security Content Update)** for prebuilt detections
* Align rules with **MITRE ATT&CK framework**

---

## ⏱️ Alert Management

* Always implement **alert throttling**
* Prevent alert storms and SOC fatigue

---

## 📊 Risk-Based Alerting (RBA)

* Assign **risk scores** to events
* Trigger alerts only when risk threshold is exceeded
